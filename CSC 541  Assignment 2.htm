Introduction

The goals of this assignment are three-fold:

  1. To investigate the use of field delimiters and record counts for field and record organization.
  2. To build and maintain an in-memory primary key index to improve search efficiency.
  3. To use an in-memory availability list to support the reallocation of space for records that are deleted.

Student File

During this assignment you will build and maintain a simple file of student records. Each record will have four fields: student ID (SID), last name (last), first name (first), and program of study (major). Fields within a record will be variable-length, and will be delimited by the "|" character. For example:

  
712412913|Healey|Christopher|CSC

represents a student with SID=712412913, last=Healey, first=Christopher, and major=CSC. To simplify processing, we assume a record will never have any spaces embedded inside it (e.g., you will never see a first like "R. Michael" or a last like "St. Amant"). This means a record can always be read as a single, indivisible string.

SID is the primary key for a student record. This means every individual student record will have a unique SID.

Records will be variable-length, and will be stored one after another in a binary data file. Each record will be preceded by an integer count (which occupies four bytes on our system). This count denotes the size of the record.

Note: Read the above description of the record count carefully! It is stored as binary data in a manner similar to how float data were stored (and retrieved) in Assignment 1. It is not appropriate to store the count as an ASCII number. For example, if you wanted to read a record at offset off in a file referenced through a filereader object fp, it would be done as:

  
#include "filereader.h"

char buffer[ 1024 ];
int count;
filereader fp;
long off;

// Open file using open() method

fp.seek( off, BEGIN );
fp.read_raw( (char *) &count, sizeof( int ) );
fp.read_raw( buffer, count );

Writing a new record would use an identical, inverse procedure. Write the integer count representing the record body's size (in bytes) in binary format using write_raw(), then format and write the body of the record, also using write_raw().

In-Memory Primary Key Index

In order to improve search efficiency, a primary key index will be maintained in memory as a reference to each record stored in the file. For our records, SID will act as a primary key. This means each entry in your index will have a structure similar to:

  
struct index_S {
  int key;     // Key of record
  long off;    // Offset of record in file
};

Index entries need to be stored in a collection that supports direct access and dynamic expansion. One good choice is a dynamically-expandable array. Index entries must be sorted by key (that is, sorted such that the smallest key is at the front of the index and the largest key is at the end). This will allow you to use a binary search to find a target key in the index.

The index will not be re-built every time the student file is re-opened, rather, it will be maintained in a permanent form on-disk. After you are finished performing operations on a student file, you will write your index to disk, saving its contents in an index file. If, at a later date, you decide to do more processing on the student file, you can open the corresponding index file, read its contents, and immediately reconstruct your primary key index. The index will have exactly the same state as before, and will be ready to use to access records in the student file.

You can use any format you want to store each key/offset pair the index file (e.g., as binary data, as ASCII data, or whatever you feel is most appropriate).

In-Memory Availability List

When records are deleted from the file, rather than closing the hole that forms (an expensive operation), we will simply record the size and the offset of the hole in an in-memory availability list. Each entry in your availability list will have a structure similar to:

  
struct avail_S {
  long off;    // Offset of hole
  int size;    // Size of hole
};

The availability list can be maintained in any order you like (i.e., it does not need to be in sorted order). The simplest technique would be to append new entries to the end of the list.

When the user tries to add a new record to the file, you should first check the availability list to see if a hole exists that is big enough to hold the new record. You can search sequentially through the availability list from start to end to look for an appropriately-sized hole. If no holes are found, the new record will be appended to the end of the file.

All of our examples assume the above strategy for maintaining the availability list: we add new entries to the end of the list, and we do a first-fit search from start to end to find an available hole. Any fragment that remains after a new entry is added to the hole (i.e., when the hole is larger than the new entry) is re-added back to the end of the availability list.

The availability list will not be re-built every time the student file is re-opened, rather, it will be maintained in a permanent form on-disk. After you are finished performing operations on a student file, you will write your availability list to disk, saving its contents in an availability file. If, at a later date, you decide to do more processing on the student file, you can open the corresponding availability file, read its contents, and immediately reconstruct your availability list. The list will have exactly the same state as before, and will be ready to use to identify holes in the student file.

You can use any format you want to store each offset/size pair in the availability file.

User Commands

The user will communicate with your program through a set of user commands typed at the keyboard. Your program should support four simple commands:

  • add rec
    Adds a new record rec to the student file. The format of rec is a |-delimited set of fields (exactly as described in Student File section above), for example:

         add 712412913|Healey|Christopher|CSC

    adds a new record with SID=712412913, last=Healey, first=Christopher, and major=CSC.

  • find key
    Finds the record with SID=key in the student file, if it exists. The record should be printed in |-delimited format, (exactly as described in Student File section above), for example:

         find 712412913

    should print on-screen

         712412913|Healey|Christopher|CSC
  • del key
    Delete the record with SID=key from the student file, if it exists.

  • end
    Ends the program, closes the student file, and writes the (possibly new) index and availability lists to the corresponding index and availability files.

Add Command

To add a new record to the student file, complete the following steps:

  1. Search the index (using a binary search) to see if an index entry exists with a key value equal to rec's SID. If such an entry does exist, then rec has the same primary key as a record already in the student file. This is not allowed, so you can ignore the add request (if the user wants to update an already-existing record, they must first delete it, then re-add it).
  2. Search the availability list for a hole large enough to hold rec (remember, the amount of space you need is the length of rec PLUS enough space to hold the count that precedes rec). If an appropriately-sized hole is found, remove it from the availability list, and write rec and its count to the hole's offset
    If the hole is bigger than the space needed to hold rec, there will be a fragment left over at the end of the hole. That fragment must be added back to the availability list as a new (smaller) hole of available space in the file.
  3. If no appropriately-sized holes exist in the file, then rec can simply be appended to the end of the file.
  4. Regardless of where rec is written, a new entry must be added to the index representing the key and offset of rec. The index must be maintained sorted by key.

Find Command

To find a record, start by searching the index (using a binary search) for an index entry with a key value equal to key. If such an entry is found, use the index entry's offset to locate and read the record, then print the record's fields on-screen.

If no index entry with the given key exists, write

  
No record with SID=key exists

on-screen.

Del Command

To delete a record, start by searching the index (using a binary search) for an index entry with a key value equal to key. If no index entry with the given key exists, write

  
No record with SID=key exists

on-screen.

If an index entry with the a key value equal to key is found, use the entry's offset to locate and read the size of the record. Since this record is being deleted, it will form a hole in the student file. Add a new entry to the availability list noting the location and size of this hole (remember, the size of the new hole is the size of the record being deleted PLUS the size of the count that precedes the record). Finally, remove the index entry corresponding to this record from the index.

End Command

This command ends the program. Be sure to close the student file, and to write the current state of your index and availability list to the corresponding index and availability files (these will be needed the next time you want to process information in the student file).

For marking purposes, whenever your program ends we want you to print the current contents of your index and availability list on-screen. For the index entries, print a line containing the text "Index:", followed by one line for each key/offset pair. Use the following format to print the index entries:

  
printf( "%d: %ld\n", index[ i ].key, index[ i ].off );

For the availability list, print a line containing the text "Availability:", followed by one line for each offset/size pair. Use the following format to print the availability list entries:

  
printf( "%d: %ld\n", avail[ i ].off, avail[ i ].size );
